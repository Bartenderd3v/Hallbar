## 1. Что такое Hall

**Hall** — прогрессивное веб-приложение (PWA) для бара Hall. Помогает гостям разобраться с меню, барменам работать быстрее и увереннее, а управляющему — понимать что происходит в баре без лишней ручной работы.

Одна ссылка — три режима работы. Открывается на любом телефоне без установки.

---

## 2. Пользователи

### 2.1 Гость
Приходит в бар. Может ничего не знать о коктейлях.

**Что ему нужно:**
- Посмотреть меню коктейлей и разобраться со вкусами
- Посмотреть меню кухни
- Посмотреть какие пластинки есть в баре

**Как попадает в приложение:** QR-код на столе → открывает меню в браузере. Никакого логина.

**Что НЕ должен видеть:** рецепты с граммовками, шаги приготовления, рабочие инструменты.

---

### 2.2 Бармен
Работает за баром. Телефон в руках — быстро, без лишних движений.

**Что ему нужно:**
- Найти рецепт коктейля за 2 секунды
- Видеть чеклист открытия и закрытия смены
- Составлять расписание выходов
- Заполнять отчёт в конце смены: что продал, чего нет в наличии, что нужно сделать завтра
- Личный профиль с метриками (сколько отдал за смену/неделю/месяц)

**Как попадает:** по ссылке-приглашению от управляющего → создаёт аккаунт → логинится.

---

### 2.3 Управляющий / Владелец
Контролирует работу бара. Смотрит на цифры, управляет командой и контентом.

**Что ему нужно:**
- Видеть аналитику: что продаётся, что нет, выручка по сменам
- Добавлять и удалять барменов
- Редактировать меню (цены, состав, описания) без кода
- Управлять чеклистами смены
- Видеть отчёты барменов

**Как попадает:** аккаунт создаётся через Supabase, роль `manager`.

## 4. Архитектура

```
GitHub Pages (index.html)
    ├── Гостевой режим (без логина)
    │     └── Загружает только публичные данные из Supabase
    │
    ├── Стафф-режим (после логина)
    │     ├── Роль: bartender → рецепты, чеклисты, профиль
    │     └── Роль: manager  → всё + админка
    │
    └── Supabase (база данных + авторизация)
          ├── auth.users — аккаунты
          ├── profiles   — имя, роль
          ├── cocktails  — рецепты (планируется)
          ├── shifts     — смены и отчёты (планируется)
          └── sales      — продажи (планируется)
```

**Стек:**
- Frontend: HTML + CSS + vanilla JavaScript (без фреймворков)
- Хостинг: GitHub Pages (бесплатно)
- База данных: Supabase PostgreSQL (бесплатный план)
- Авторизация: Supabase Auth

## 5. Безопасность

### Текущие проблемы
- [ ] Рецепты с граммовками хранятся в JS-коде — видны в исходнике
- [ ] Самостоятельная регистрация отключена? (нужно проверить)

### Принципы
- Гость не видит граммовки и шаги приготовления — ни в интерфейсе, ни в коде
- Рецепты загружаются из базы только после авторизации
- RLS-политики на всех таблицах с данными
- Anon key в клиентском коде — это нормально по дизайну Supabase
- Пароли барменов минимум 8 символов

## 7. Правила работы над проектом

### Как мы работаем с задачами
1. **Одна задача за раз.** Не начинаем следующую пока не проверили предыдущую.
2. **Сначала спецификация фичи, потом код.** Перед каждой новой функцией — короткое описание: что делает, кто видит, как проверить.
3. **Проверка на телефоне** после каждого деплоя. Не только на компьютере.
4. **Коммиты с понятными сообщениями:** `feat: кнопка отдал`, `fix: звёздочки крепости`, `refactor: рецепты в базу`.

### Шаблон задачи (перед написанием кода)
```
## Задача: [название]

**Кто пользуется:** гость / бармен / управляющий
**Что должно работать:**
- пункт 1
- пункт 2
**Что НЕ входит в задачу:**
- пункт
**Как проверить что готово:**
- открыть на телефоне и ...
```

## 8. Глоссарий

| Термин | Значение |
|---|---|
| PWA | Progressive Web App — сайт который работает как приложение |
| RLS | Row Level Security — защита строк в базе данных |
| Supabase | База данных + авторизация (бесплатный облачный сервис) |
| GitHub Pages | Бесплатный хостинг для статичных сайтов |
| Anon key | Публичный ключ Supabase — безопасно хранить в коде |
| Стафф-режим | Интерфейс для сотрудников бара после авторизации |
